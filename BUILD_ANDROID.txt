بناء APK — توقعات واقعية
========================

1) لا يوجد ضمان "صفر أخطاء" قبل أول بناء فعلي. أول APK غالباً يحتاج تعديل إصدارات NDK أو المتطلبات.

2) macOS: Buildozer الرسمي يستهدف لينكس. الأسهل:
   - تشغيل أوبنتو (جهاز/VM/WSL2) أو
   - GitHub Actions (ملف .github/workflows/android-apk.yml) أو
   - حاوية Docker إن توفرت صورة buildozer متوافقة مع مشروعك.

3) tflite-runtime على python-for-android قد يفشل (رابط لمناقشات: https://github.com/kivy/python-for-android/issues/3191 ).
 إن فشل البناء عند tflite-runtime:
   - مؤقتاً احذف tflite-runtime من سطر requirements في buildozer.spec
   - أعد البناء؛ التطبيق سيعمل لكن تحليل الصور عبر النموذج لن يعمل حتى تضيف حلاً (وصفة أخرى / دمج مكتبة).

4) بعد البناء: الملف غالباً bin/watermonitor-*.apk

5) ثبّت على الهاتف "تثبيت من مصادر غير معروفة" أو عبر adb install.

6) ضع humidity_model.tflite و labels.txt و logo.png و alarm.wav في المشروع قبل البناء إن أردتها داخل الـ APK (source.include_exts يشملها).

7) إن ظهر "Aidl not found" أو "license is not accepted" لـ build-tools;37: Buildozer يستخدم ~/.buildozer/.../android-sdk. الـ workflow الحالي يقبل التراخيص عبر yes | sdkmanager --licenses، يثبت build-tools، ويربط مجلد Buildozer رمزياً بنفس ANDROID_SDK_ROOT.

8) إن ظهر "sdkmanager path .../tools/bin/sdkmanager does not exist": تمت إضافة رابط رمزي من tools/bin/sdkmanager إلى cmdline-tools (مطلوب لبعض إصدارات p4a).
